Design Tokens

What is a design token?

Design token is a named entity that store visual design attributes in an agnostic, human-readable abstraction of visual styles that sync with all the style files in the system. They allows us to create a scalable and consistent visual system for UI development.

Why use a design token?

Design tokens are versatile

Design tokens are platform-agnostic human-readable names instead of using them as hard-coded values like hex codes, pixel values, etc. This means they are independent of any specific platform or tech stack, making it easy to implement them across platforms and technologies.

Three tier token structure

The three layers consist of base tokens, alias tokens and component tokens each with more decisions and granularity built in and all referencing from each other.

Token Strategy

By simply observing these tokens, both designers and developers can intuitively discern its purpose and usage, without diving into its underlying value or referring to documentation.

Currently I have a challenge at hand which involves optimizing a design system with existing tokens. I have identified opportunities to enhance efficiency. This includes the organization of components through grouping categorically.

Assigning each atom and component to a specific category facilitates the application of these categories in our token naming convention. It's important to note that this categorization approach does not necessarily impact the atomic design structure of the overall design system.

Benefits

  • Consolidating repetitive tokens into a single category will significantly decrease the overall token count.
  • Better manageability and clear visual hierarchies, gives lesser cognitive load.
  • Fewer tokens in the library, reducing token inventory count by fifty percent.
  • Enhance control over agency operations with standardized tokens.

Token Taxonomy

Efficiently deploying design tokens is crucial, as the process can quickly become unwieldy through token generation, leading to a rise in token inventory.

Token taxonomy expanded

Base and alias tokens

Each alias token is designated with a distinct purpose, meaning, and function. Unlike base tokens, alias tokens are exclusively used within predefined contexts or instances.

Usage

The naming conventions for alias tokens should strike a balance between abstraction to accommodate various values and clarity to effectively convey their intended use or purpose.

Token updates and isolation

The hierarchical structure of design tokens with multiple tiers facilitates the concept of isolation and control within a design system.

Token updates and isolation on base token

Base tokens serve as repositories for universal choices, encompassing higher-level design decisions. They lack isolation, meaning that any modifications made to global tokens will impact a wide array of design choices throughout a design system.

Token updates and isolation on alias token

Alias tokens possess distinct intent and significance. Positioned as mid-level design choices, they offer a moderate degree of isolation. Modifying an alias token will manifest changes within defined contexts or instances.

Token updates and isolation on component token

Tokens specific to components are established at the component level and exhibit the highest level of isolation. When you update a component-specific token, it exclusively affects that particular component and no others.

Token pipeline and workflow

The process involves utilising the open-source Figma Token Studio plugin to generate tokens and exporting to GitHub in JSON format, where the plugin automatically converts the tokens JSON. Subsequently, the tokens in JSON format are analyzed through Style Dictionary on GitHub and converted to CSS variables to use them code.
